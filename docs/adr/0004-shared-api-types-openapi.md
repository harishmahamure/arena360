# ADR-0004: Shared API types generated from backend OpenAPI

**Status**: Accepted
**Date**: 2026-05-27
**Deciders**: Platform team (gaming-cafe consolidation working group)

## Context

Today, each frontend hand-writes its own response and request types
for the backend's REST surface:

- `game-zone-management-fe/apps/admin/src/services/**/*.ts` — admin's
  duplicated response shapes (one per service file).
- `agent-kiosk/src/shared/types/api.ts` — kiosk's monolithic type
  file, plus a duplicate `agent-kiosk/open-api.json` checked in to the
  kiosk repo and used for ad-hoc validation.

This produces three concrete problems:

1. **Drift.** When the backend changes a field name, no compiler error
   surfaces in admin or kiosk until the runtime breaks at the user's
   screen.
2. **Duplication.** The same `Plan`, `Session`, `Device` shapes are
   written two or three times across the repos.
3. **No single source of truth.** The kiosk's `open-api.json` is a
   month-stale copy of the backend's; nobody is sure which is correct.

The backend already produces an OpenAPI spec via
`arena360-backend/scripts/generate-openapi.ts`, emitting
`arena360-backend/docs/openapi.json`. We have a generator in the
ecosystem (`openapi-typescript`) that can turn that into TypeScript
types with zero runtime footprint.

## Decision

We will treat the **backend's OpenAPI spec as the single source of
truth** for HTTP request and response shapes, and generate a shared
`@gaming-cafe/api-types` package from it.

**Note**: Runtime values (enums like `ErrorCode`, constant maps like
`ErrorMessages`) live in `@gaming-cafe/contracts`, not here. This
package is regenerated by `openapi-typescript` on every CI run, so any
hand-written code would be overwritten. See ADR-0008 for the contracts
package rationale.

Concretely:

1. **Generator pipeline**:

   ```
   pnpm gen:api-types
     │
     ├── runs apps/backend/scripts/generate-openapi.ts
     │     → writes apps/backend/docs/openapi.json
     │
     ├── runs openapi-typescript apps/backend/docs/openapi.json \
     │     -o packages/api-types/src/schema.ts
     │
     └── runs pnpm --filter @gaming-cafe/api-types build
   ```

   The root `package.json` declares the `gen:api-types` script; the
   sub-scripts live in the relevant workspaces.

2. **`@gaming-cafe/api-types` public surface**:

   ```ts
   export type { paths, components, operations } from './schema';

   // Convenience aliases used in app code:
   export type PathsResponse<P extends keyof paths, M extends 'get'|'post'|'put'|'patch'|'delete'> =
     paths[P][M] extends { responses: { 200: { content: { 'application/json': infer R } } } } ? R : never;

   export type PathsRequest<P extends keyof paths, M extends 'get'|'post'|'put'|'patch'|'delete'> =
     paths[P][M] extends { requestBody: { content: { 'application/json': infer R } } } ? R : never;

   export type Schema<K extends keyof components['schemas']> = components['schemas'][K];
   ```

   That is the entire surface. No hand-edited types live in this
   package.

3. **Consumers**:

   - `apps/admin` deletes its hand-written response interfaces and
     imports from `@gaming-cafe/api-types`.
   - `apps/kiosk` deletes `src/shared/types/api.ts` and the duplicated
     `open-api.json`, importing from `@gaming-cafe/api-types`.
   - The shared HTTP client in `packages/utils` is parametric over
     the schema types so each call site stays end-to-end typed.

4. **Drift check in CI**. A CI job runs `pnpm gen:api-types` and then
   `git diff --exit-code packages/api-types/src/schema.ts` (and
   `apps/backend/docs/openapi.json`). Any difference fails the build.
   The fix is local: re-run `pnpm gen:api-types` and commit the diff.

## Consequences

### Positive

- **One source of truth.** The OpenAPI spec emitted from Nest
  decorators is canonical; clients can never disagree with the server.
- **Compile-time drift detection.** Renaming a field in a DTO breaks
  the admin and kiosk builds the next time CI runs `pnpm gen:api-types`
  — no surprise at runtime.
- **Less hand-written code.** Entire files of response interfaces are
  deleted.
- **Zero runtime cost.** `openapi-typescript` emits types only; no
  client code, no bundled OpenAPI document.
- **Tooling-friendly.** IDE auto-completion on `paths['/sessions']['post']`
  is excellent and immediate.

### Negative

- **Coupling to OpenAPI accuracy.** If a Nest endpoint forgets its
  `@ApiResponse(...)` decorator, the generated types miss it and the
  drift check passes a wrong-shape contract. Mitigation: lint rule
  (or PR review checklist) requiring `@ApiResponse` on every
  controller method; we can add a custom Nest schematic later.
- **Two-step change** when both backend and frontend evolve in the
  same PR: backend change first → `pnpm gen:api-types` → frontend
  uses the new types. Acceptable; it's the same workflow as any
  generated client.
- **`packages/api-types/src/schema.ts` is a tracked generated file.**
  Some teams prefer ignoring generated artefacts. We track it so
  consumers can install the package straight from git without
  running the generator, and so CI's drift check has a baseline.

### Risks

- **Risk**: developer forgets `pnpm gen:api-types` and pushes drifted
  types.
  **Mitigation**: CI drift check; a pre-push git hook running the
  same check.
- **Risk**: backend boots its full Nest application (with DB) just to
  emit the OpenAPI spec, making `pnpm gen:api-types` slow.
  **Mitigation**: `generate-openapi.ts` already short-circuits the
  Nest factory to skip DB connect on `process.env.OPENAPI_ONLY=1`;
  verified by the backend team.
- **Risk**: `openapi-typescript` upgrade changes the generated output
  shape and breaks consumers.
  **Mitigation**: pin the version in the root `package.json`;
  upgrade in a dedicated PR so the diff is reviewable.

## Alternatives considered

### Hand-written `@gaming-cafe/api-types` (no generator)

- Pros: no generator dependency; types can be more ergonomic than the
  literal OpenAPI projection.
- Cons: drift returns immediately; defeats the purpose of the package.
- **Why rejected**: not solving the original problem.

### tRPC or GraphQL codegen

- Pros: end-to-end type safety with zero hand-written types.
- Cons: requires rewriting the backend's HTTP surface; tRPC couples
  client and server tightly; GraphQL adds a new transport and
  schema language.
- **Why rejected**: out of scope for the consolidation; revisit if we
  re-platform the backend.

### OpenAPI Generator (Java-based) producing a typed client

- Pros: produces a full SDK (types + runtime client) for many
  languages.
- Cons: Java toolchain dependency; generated SDK is heavy and
  opinionated; we already have a working axios setup we want to
  consolidate, not replace.
- **Why rejected**: too much weight for a JS-only consumer set.

### Per-consumer `openapi-typescript` invocation (no shared package)

- Pros: marginally simpler — each app generates its own types.
- Cons: duplicates the schema file in two apps; loses the package
  boundary that lets us evolve the helpers in one place; harder to
  add a drift check.
- **Why rejected**: a single shared package is strictly better.

## Implementation notes

- `packages/api-types/package.json` declares
  `"openapi-typescript": ">=7"` as a devDep; the runtime package has
  no deps (it ships pure types).
- The root `package.json` adds `"gen:api-types": "pnpm --filter
  @gaming-cafe/backend openapi:emit && openapi-typescript
  apps/backend/docs/openapi.json -o packages/api-types/src/schema.ts
  && pnpm --filter @gaming-cafe/api-types build"`.
- CI runs `pnpm gen:api-types` and `git diff --exit-code` early in
  the pipeline, before the test matrix, so a drift failure is fast.
- The migration of consumers (admin + kiosk) is the
  `api-types-package` todo in the consolidation plan, sequenced after
  the moves in Phase 2.
